# Панель «Свойства» справа (отдельная панель)

## Текущее состояние

- **Правая зона** ([FormPreviewApp.tsx](src/webview/components/FormPreview/FormPreviewApp.tsx) ~1434–1477): одна панель с вкладками «Свойства», «Реквизиты», «Команды», «Параметры». При выборе «Свойства» в теле панели рендерится [ElementPropertiesEditor](src/webview/components/FormPreview/FormPreviewApp.tsx) (таблица свойство/значение/тип).
- **Кнопка «Свойства»**: в дереве элементов уже есть кнопка с иконкой sequence.svg; по клику вызывается `onSelect(path)` и `setRightTab('properties')` (стр. ~1319–1322).

## Целевое поведение

- Вкладки справа остаются только: **Реквизиты**, **Команды**, **Параметры** (вкладку «Свойства» убрать).
- По нажатию кнопки «Свойства» у элемента открывается **отдельная панель справа** (третья колонка в `edt-top`), а не переключение вкладки.
- Панель содержит (по скрину):
  - Заголовок: «Свойства: Поле» (или тип элемента, например InputField → «Поле ввода») и кнопка закрытия (X).
  - Строка поиска: «Поиск (Ctrl+Alt+I)» и при необходимости иконки (сортировка, фильтр) — на первом этапе достаточно поиска по свойствам.
  - Секция **«Основные»**: сворачиваемая; содержимое — текущий редактор свойств (сетка свойство/значение/тип), по сути нынешний `ElementPropertiesEditor`.
  - Секция **«События»**: сворачиваемая; список событий элемента (ПриИзменении, НачалоВыбора и т.д.) с полем ввода/кнопкой «…». MVP: показывать события из `item.properties.Events` или статичный список известных событий для типа элемента; обработчики — поле ввода или заглушка.

## План реализации

### 1. Состояние и открытие панели

- Ввести в `FormPreviewApp` состояние:
  - `propertiesPanelOpen: boolean` (по умолчанию `false`).
- Обработчик кнопки «Свойства» в дереве: вызывать `onSelect(path)` и `setPropertiesPanelOpen(true)` (вместо `setRightTab('properties')`).
- Вкладку «Свойства» убрать из `PaneTabs`; при открытии панели вкладку справа не переключать (или оставить текущую). Значение по умолчанию для `rightTab` задать, например, `'attributes'`.

### 2. Разметка: третья колонка (панель свойств)

- В `.edt-top` после текущей правой панели (и при необходимости после вертикального ресайзера) рендерить панель свойств только при `propertiesPanelOpen`.
- Структура:
  - Опционально вертикальный разделитель (resizer) между средней панелью и панелью свойств, с сохранением ширины панели свойств в state (по аналогии с `leftPaneWidthPercent`).
  - Контейнер панели: заголовок, тело с поиском и секциями.
- Ширину панели свойств задать фиксированной (например 320px) или в процентах с возможностью ресайза; минимальная ширина ~260px.

### 3. Компонент панели «Свойства»

- Вынести или оформить как подкомпонент в том же файле:
  - **Шапка**: текст «Свойства: &lt;тип/название&gt;» (тип из `selectedItem.type` или человекочитаемое имя, например InputField → «Поле ввода»), кнопка закрытия (X), по клику — `setPropertiesPanelOpen(false)`.
  - **Поиск**: поле «Поиск (Ctrl+Alt+I)»; фильтрация списка свойств в секции «Основные» по введённому тексту (по имени свойства).
  - **Секция «Основные»**: `<details>` или кастомный сворачиваемый блок, по умолчанию открыт; внутри — текущий контент `ElementPropertiesEditor` (таблица свойств выбранного элемента + сохранение через существующий `onChange`). Если элемента нет — показывать «Выберите элемент в дереве» или скрывать панель.
  - **Секция «События»**: `<details>` или кастомный блок; внутри — список событий. Данные: из `item.properties.Events` (если есть в форме) или статичный список для типа (например для InputField: ПриИзменении, НачалоВыбора, Очистка, Открытие и т.д.). Каждая строка: имя события + поле ввода (или кнопка «…» для выбора обработчика). MVP: отображение и, по возможности, редактирование строки обработчика без полноценного открытия модуля.

### 4. Стили (editor.css)

- Классы для панели свойств: обёртка (например `.edt-properties-panel`), заголовок с кнопкой закрытия, строка поиска, блоки секций. Оформление в духе существующих `.edt-pane`, `.edt-toolbar`, чтобы визуально соответствовать скрину (отдельная «полоса» справа).

### 5. События (Events) — источник данных

- В XML формы события элемента могут быть в `item.properties.Events` (массив/объект с полями Event). При наличии — парсить и выводить в секции «События».
- Если в текущем парсере структура Events не приходит — ограничиться заголовком секции и заглушкой «Нет событий» или статичным списком типовых имён событий для выбранного типа элемента без привязки к XML (только UI по скрину).

## Риски и упрощения

- Полная привязка событий к обработчикам в коде формы выходит за рамки MVP; достаточно отобразить список событий и возможность редактировать значение (например имя процедуры) в state/XML при наличии парсинга Events.
- Ресайз панели свойств опционален; можно зафиксировать ширину (например 320px) и добавить ресайзер позже.
