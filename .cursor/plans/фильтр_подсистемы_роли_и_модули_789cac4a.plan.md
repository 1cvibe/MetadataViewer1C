# Фильтрация ролей и общих модулей по подсистеме

## Проблема

При установке фильтра по подсистеме роли и общие модули не фильтруются — отображаются все объекты вместо только тех, что входят в Content выбранной подсистемы.

## Анализ

**Поток данных:**

1. Пользователь выбирает подсистему → `filterBySubsystem` получает `commandArguments` = результат `CollectSubsystemContent(rootPath, treeItemPath)`
2. `CollectSubsystemContent` читает XML подсистемы (например, `Subsystems/ibs_РасчетПоказателейПремирования.xml`) и извлекает Content
3. Content содержит элементы вида `<xr:Item xsi:type="xr:MDObjectRef">Role.ibs_ДобавлениеИзменениеТипыПремий</xr:Item>`
4. Список объектов передаётся в `subsystemFilter` и используется в `CreateTreeElements` / `Edt.createTreeElements`
5. В reduce: `if (subsystemFilter.length && subsystemFilter.indexOf(current.$_name) === -1) return previous` — объект исключается, если его имя не в списке

**Возможные причины:**

1. **Парсинг Content в CollectSubsystemContent** — fast-xml-parser может возвращать разную структуру (namespace, ключи, расположение текста)
2. **Формат treeItemPath** — для вложенных подсистем путь может формироваться неверно, из-за чего читается не тот файл или файл не находится
3. **EDT-путь** — при использовании Configuration.mdo формат имён объектов может отличаться

## Изменения

### 1. Усилить парсинг Content в CollectSubsystemContent

**Файл:** [src/metadataView.ts](src/metadataView.ts), функция `CollectSubsystemContent` (стр. 2844–2872)

Добавить дополнительные варианты извлечения текста из элементов Content:

- Проверять `contentElem["@_xsi:type"]` (атрибут с префиксом)
- Обрабатывать случай, когда текст в корне объекта (некоторые парсеры возвращают `{ "xr:MDObjectRef": "Role...." }`)
- Добавить fallback: если объект имеет одно свойство-строку (не `#text`, не `$_*`), использовать его как текст
- Явно обрабатывать `contentElem.text` и `contentElem.value` (уже есть, но проверить порядок)

```ts
// Расширенное извлечение текста
if (typeof contentElem === 'string') {
  text = contentElem;
} else if (contentElem && typeof contentElem === 'object') {
  text = contentElem["#text"] ?? contentElem.text ?? contentElem.value ?? 
    (() => {
      const keys = Object.keys(contentElem).filter(k => !k.startsWith('@') && !k.startsWith('$_'));
      if (keys.length === 1 && typeof contentElem[keys[0]] === 'string') {
        return contentElem[keys[0]];
      }
      return null;
    })();
}
```

### 2. Добавить варианты ключей для Content

**Файл:** [src/metadataView.ts](src/metadataView.ts), стр. 2848–2850

Проверять дополнительные ключи для доступа к элементам Content (в зависимости от парсера и namespace):

```ts
content = contentNode["xr:Item"] || contentNode.Item || 
  contentNode["xr:content"] || contentNode.content ||
  (Array.isArray(contentNode) ? contentNode : null);
```

### 3. Включить preserveOrder / ignoreNamespace для парсера подсистем

**Файл:** [src/metadataView.ts](src/metadataView.ts), стр. 2835–2839

Убедиться, что парсер не теряет текст. При необходимости добавить `textNodeName: "#text"` (если поддерживается) или `ignoreNamespace: false` для сохранения префиксов.

### 4. Проверка EDT-пути (если используется)

**Файл:** [src/ConfigurationFormats/edt.ts](src/ConfigurationFormats/edt.ts), стр. 58–61

В EDT `obj` берётся из `configuration[objectsName]`. Убедиться, что формат совпадает с `subsystemFilter` (например, `CommonModule.Имя` vs `commonModule.Имя`). При несовпадении — нормализовать перед сравнением.

## Диагностика

Включить `metadataViewer.debugMode` и проверить логи:

- `[CollectSubsystemContent] Объекты: ...` — что реально извлекается из XML
- Сравнить с именами в ConfigDumpInfo (`Role.*`, `CommonModule.*`)

## Риски

- Разные версии fast-xml-parser могут давать разную структуру
- Вложенные подсистемы (например, `Subsystems/A/Subsystems/B`) — путь к файлу может отличаться
