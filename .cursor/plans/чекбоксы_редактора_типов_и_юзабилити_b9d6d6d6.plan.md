# Анализ чекбоксов в редакторе типов и улучшения юзабилити

## 1. Причина некорректного отображения чекбоксов

**Симптом:** В модальном окне «Редактор типа» (TypeWidget) чекбоксы отображаются **сверху** от текста типа вместо **слева**, что нарушает ожидаемый UI.

**Корневая причина — переопределение стилей по каскаду:**

- Элементы списка типов — это `<label className="type-item">` с `<input type="checkbox">` и `<span>{label}</span>` ([TypeWidget.tsx](src/webview/widgets/TypeWidget.tsx) ~1486, 1532, 1626).
- Для них заданы нужные стили в [editor.css](src/webview/styles/editor.css) (строки 420–443):
  - `.type-item { display: flex; flex-direction: row; align-items: center; gap: 8px; }`
  - `.type-item input[type="checkbox"] { flex-shrink: 0; order: -1; }`
- TypeWidget используется внутри **PredefinedTypeEditorModal** и **CharacteristicTypeEditorModal**, которые оборачивают его в `<div className="form-field">` ([PredefinedTypeEditorModal.tsx](src/webview/components/PredefinedEditor/PredefinedTypeEditorModal.tsx) 163–165, [CharacteristicTypeEditorModal.tsx](src/webview/components/FormEditor/CharacteristicTypeEditorModal.tsx) 89–91).
- Модальное окно TypeWidget (`type-editor-modal-overlay` > `type-editor-modal`) рендерится **внутри** этого же `form-field` (как потомок типа form-field > … > TypeWidget > type-editor-modal).
- В [editor.css](src/webview/styles/editor.css) (1091–1096) задано:
  ```css
  .form-field label {
    display: block;
    margin-bottom: 6px;
    ...
  }
  ```

- Специфичность: `.form-field label` (класс + элемент) **выше**, чем у `.type-item` (один класс). В итоге ко **всем** `label` внутри формы, в т.ч. к `label.type-item` в редакторе типов, применяется `display: block`.
- При `display: block` флекс-раскладка `.type-item` не используется, содержимое ведёт себя как обычный поток: чекбокс и текст переносятся/выстраиваются вертикально, визуально чекбокс оказывается «сверху» текста.

**Аналогично** страдают заголовки категорий «Примитивные типы», «Специальные типы», «Ссылочные типы»: они тоже `<label>` внутри `form-field` и попадают под `.form-field label`.

## 2. Варианты исправления чекбоксов

**Вариант A (рекомендуется): ужесточить область действия `.form-field label`**

- Ограничить стили только **прямым** потомком: `.form-field > label`.
- Тогда «Тип» и другие подписи полей в модалках останутся `display: block`, а вложенные `label` внутри TypeWidget (в т.ч. `.type-item` и заголовки категорий) перестанут переопределяться.
- Файл: [editor.css](src/webview/styles/editor.css). Правка в блоке ~1091–1096.
- Нужно проверить все использования `form-field`: в модалках (AddTabularAttributeModal, AddAttributeToObjectModal, AccountingFlagEditorModal и т.д.) — чтобы нигде не использовались вложенные `label` как подписи полей внутри `form-field`. По текущей разметке везде только `form-field > label` + инпут/виджет, изменение безопасно.

**Вариант B: повысить специфичность селекторов редактора типов**

- Использовать контекстные селекторы, например:
  - `.type-editor-left .type-item`
  - `.type-categories .type-item`
  - `.type-category-header label` оставить как есть или тоже усилить через `.type-categories .type-category-header label`.
- Так мы явно задаём приоритет стилей для списка типов над `.form-field label`.
- Минус: не убирает глобальное переопределение `label` внутри `form-field`, только «отбиваем» конкретные блоки.

**Вариант C: не использовать `label` для пунктов списка типов**

- Заменить `label` на `div` с `role="button"`, `tabIndex`, `onClick` и `aria-checked` и стилизовать как строку с чекбоксом.
- Чекбоксы гарантированно перестанут попадать под `.form-field label`, но потребуются доработки доступности и кликабельности всей строки.

Оптимально — **вариант A** как минимальное и целенаправленное изменение с поправкой на логику «form-field = контейнер поля с одной подписью».

## 3. Дополнительные правки для согласованности (после A)

- Явно задать для элементов списка и заголовков категорий:
  - `.type-item`: оставить `display: flex`, `flex-direction: row`, `align-items: center`, чекбокс слева.
  - `.type-category-header label`: проверить, что остаётся `display: flex`, `align-items: center`, чекбокс слева от «Примитивные типы» и др.
- Убедиться, что при открытии редактора типов из PredefinedTypeEditorModal и CharacteristicTypeEditorModal чекбоксы во всех трёх блоках (примитивные, специальные, ссылочные) отображаются **слева** от текста.

## 4. Дублирование правил в editor.css

- Классы `.type-category`, `.type-list` объявлены **дважды** (блоки ~387–418 и ~633–648).
- Второй блок частично переопределяет первый (`margin-bottom` у `.type-category`, один и тот же `.type-list`).
- Стоит **объединить** правила в один блок и убрать дубликаты, чтобы не было путаницы и неожиданных переопределений при дальнейших правках.

## 5. Рекомендуемые улучшения юзабилити редакторов

Краткий список направлений (без жёсткой очередности):

| Область | Что сделать |

|--------|--------------|

| **Редактор типа (TypeWidget)** | Фикс чекбоксов (п. 2–3). Единый вид чекбоксов в категориях и в группах «СправочникСсылка» и т.д. |

| **Клавиатура** | Закрытие модалок по Escape (где ещё не реализовано). Поддержка Enter для «Сохранить» в формах (в т.ч. редактор типа, модалки предопределённых данных). Tab-навигация по полям и кнопкам без «провалов» фокуса. |

| **Фокус** | При открытии модалки фокус на первое значимое поле (поиск в редакторе типа, первое поле в модалках добавления/редактирования). Опционально — фокус-ловушка в модалках. |

| **Кнопки и иконки** | У всех иконок (например, ✎ «Открыть редактор типов») — `title`/`aria-label` с понятным описанием действия. |

| **Ошибки и валидация** | Явное отображение ошибок валидации у полей (например, «Имя»/«Код» в предопределённых данных), без использования `alert`. |

| **Единообразие модалок** | Типовой набор: заголовок, кнопки «Отмена»/«Сохранить» в одном и том же месте и порядке. Где уже используется [Modal](src/webview/components/FormEditor/Modal.tsx) — придерживаться его структуры; TypeWidget пока использует свою модалку — по возможности унифицировать. |

| **Списки и сетки** | В редакторе предопределённых данных проверка, что карточки в grid (`.attributes-list`) стабильно в несколько колонок, «Элементы (N)» в шапке и т.д. |

| **Доступность** | `aria-label` у интерактивных элементов без видимой подписи; `role` у кастомных «чекбоксов»/«кнопок», если оставляем вариант C. |

Эти пункты можно разнести по отдельным задачам и реализовывать постепенно после исправления чекбоксов.

## 6. План действий (реализация)

1. **Исправить чекбоксы**

   - В [editor.css](src/webview/styles/editor.css): заменить `.form-field label` на `.form-field > label` (ориентировочно строки 1091–1096).
   - Пересобрать webview (`npm run build:webview`), проверить редактор типа из PredefinedTypeEditorModal и CharacteristicTypeEditorModal: чекбоксы слева от текста во всех категориях.

2. **Проверить категории**

   - Убедиться, что «Примитивные типы», «Специальные типы», «Ссылочные типы» и пункты в группах (СправочникСсылка и т.д.) выглядят единообразно, чекбокс слева.

3. **Привести в порядок CSS**

   - Объединить дублирующиеся правила `.type-category` / `.type-list` в [editor.css](src/webview/styles/editor.css), удалить лишние объявления.

4. **Дальнейшие улучшения**

   - Выбрать из п. 5 приоритетные пункты (например, Escape/Enter, фокус при открытии, `title` у кнопок) и оформить отдельными задачами.

## 7. Риски и ограничения

- **Вариант A:** если где-то сознательно используется вложенная структура `form-field > div > label` для подписи поля, ограничение `> label` может это затронуть. По текущему коду такие случаи не обнаружены.
- Изменение лишь CSS, логика TypeWidget и модалок не меняется. Регрессии маловероятны при аккуратной правке селекторов и проверке всех модалок с `form-field`.
