# Анализ поведения fast-xml-parser и fast-xml-builder

## Выводы из экспериментов

### 1. Поведение парсера (XMLParser)

#### При `attributeNamePrefix: ""` (текущая настройка)
- **Проблема**: Парсер **НЕ РАЗЛИЧАЕТ** атрибуты и элементы
- Оба типа сохраняются как простые свойства объекта
- Пример:
  - XML: `<Properties Name="Test">` → `{ Properties: { Name: "Test" } }`
  - XML: `<Properties><Name>Test</Name></Properties>` → `{ Properties: { Name: "Test" } }`
- **Результат**: Информация о том, что было атрибутом, а что элементом, **теряется**

#### При `attributeNamePrefix: "@"` или `"@_"`
- Атрибуты маркируются префиксом: `{ Properties: { "@Name": "Test", Synonym: "Value" } }`
- Элементы остаются без префикса
- **Преимущество**: Можно различить атрибуты и элементы
- **Недостаток**: Требует изменений во всем коде, работающем с парсером

### 2. Поведение билдера (XMLBuilder)

#### Правила преобразования объектов в XML:

1. **Простые свойства** (строки, числа, булевы) → **атрибуты**
   ```javascript
   { Properties: { Name: "Test" } }
   → <Properties Name="Test"></Properties>
   ```

2. **Объекты с `#text`** → **элементы**
   ```javascript
   { Properties: { Name: { "#text": "Test" } } }
   → <Properties><Name>Test</Name></Properties>
   ```

3. **Объекты (не примитивы)** → **элементы**
   ```javascript
   { Properties: { Synonym: { "v8:item": {...} } } }
   → <Properties><Synonym>...</Synonym></Properties>
   ```

4. **Пустые значения**:
   - Пустая строка `""` → атрибут: `<Properties Comment=""></Properties>`
   - Объект `{ "#text": "" }` → элемент: `<Properties><Comment></Comment></Properties>`

5. **Опция `alwaysCreateTextNode: true`**:
   - Не влияет на простые свойства - они все равно становятся атрибутами
   - Влияет только на обработку текстовых узлов внутри элементов

### 3. Проблемные места в текущем коде

#### `parseProperties` (metadataParser.ts)
- **Строка 311-313**: Простые строковые значения сохраняются как строки
  - Теряется информация о том, были ли они атрибутами или элементами
- **Строка 314-316**: Булевы значения преобразуются в строки
  - Теряется информация о структуре
- **Строка 278-310**: Сложные объекты очищаются от префиксов
  - Может теряться информация о структуре, если объект был атрибутом

#### `handleSave` (MetadataPanel.ts)
- Использует нормализованные данные из `parseProperties`
- Не имеет доступа к исходной структуре XML
- Вынужден "угадывать" формат при сохранении

## Решение

### Стратегия: Сохранение исходного объекта из парсера

**Вариант A** (рекомендуемый): Сохранять исходный объект из парсера вместе с нормализованным
- При парсинге сохранять исходный объект в поле `_raw` или `_original`
- При сохранении использовать исходный объект для определения структуры
- Обновлять только измененные свойства, остальные брать из исходного объекта

**Преимущества**:
- Полная информация о структуре
- Минимальные изменения в существующем коде
- Можно постепенно мигрировать

**Недостатки**:
- Увеличение использования памяти (дублирование данных)
- Нужно аккуратно объединять изменения с исходной структурой

### Альтернативный вариант: Использовать `attributeNamePrefix: "@"`

**Преимущества**:
- Явное различие атрибутов и элементов
- Компактное хранение

**Недостатки**:
- Требует изменений во всем коде, работающем с парсером
- Нужно обновить все места, где читаются свойства (например, `obj.Properties.Name` → `obj.Properties["@Name"]`)

## Рекомендации

1. Использовать **Вариант A**: сохранять исходный объект из парсера
2. Создать утилиту для объединения изменений с исходной структурой
3. При сохранении:
   - Брать структуру из исходного объекта
   - Обновлять только измененные свойства
   - Использовать `{ "#text": value }` для элементов, простые значения для атрибутов
